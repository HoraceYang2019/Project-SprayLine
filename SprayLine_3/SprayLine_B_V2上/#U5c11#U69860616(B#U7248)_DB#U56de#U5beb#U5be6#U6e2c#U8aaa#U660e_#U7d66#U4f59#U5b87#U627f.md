# 少榆0616ver_4：Database/versionB 回寫實測說明

這份說明是給余宇承協助測試少榆端 `少榆0616ver_4` 是否能正確接上 `Database/versionB`，並完成資料庫回寫。

## 一、測試目標

請協助確認少榆端能完成以下流程：

```text
Database/versionB sensor_1min / sensor_3min
-> 少榆 MonitoringWorker 查詢感測資料
-> threshold 判斷 warning / fault
-> 寫入 alert_event
-> 建立 alert_cause_link / alert_response_link
-> 更新 batch_station_status
-> FutureService 寫入 future_prediction_result
```

這次測試重點是「少榆端是否能直接 import Database/versionB function，並完成 DB 回寫」，不是測 HTTP API，也不是測 UI。

---

## 二、檔案放置位置

請將 `少榆0616ver_4` 放在 GitHub 專案內：

```text
Project-SprayLine-main/
├─ Database/
│  └─ versionB/
└─ SprayLine_3/
   └─ 少榆0616ver_4/
```

建議工作目錄切到：

```powershell
cd "C:\你的路徑\Project-SprayLine-main\SprayLine_3\少榆0616ver_4"
```

## 三、環境變數設定

請在 PowerShell 設定以下環境變數。

`SPRAYLINE_PROJECT_ROOT` 必須指向 `Project-SprayLine-main` 根目錄，不是指向 `少榆0616ver_4`。

```powershell
$env:SPRAYLINE_PROJECT_ROOT="C:\你的路徑\Project-SprayLine-main"

$env:DB_HOST="localhost"
$env:DB_PORT="5432"
$env:DB_USER="postgres"
$env:DB_PASSWORD="你的PostgreSQL密碼"
$env:DB_NAME="sprayline"

$env:SPRAYLINE_MONITOR_STATIONS="Station_1"
$env:SPRAYLINE_MONITOR_LOOKBACK_MINUTES="10"
$env:SPRAYLINE_DUPLICATE_ALERT_SUPPRESSION_MINUTES="5"
```

如果 PostgreSQL 不在本機，請把 `DB_HOST` 改成實際 IP。

請不要把 DB 密碼寫進 GitHub、README 或任何提交檔案。

## 五、第一步：確認少榆端抓到正式 Database/versionB

執行：

```powershell
python -m webservices.integration_adapter.database_versionb_adapter
```

期待結果：

```text

http_endpoint_used = false

```

若 `database_versionB_path` 指到：

```text
...\少榆0616ver_4\external\Database\versionB
```

代表 `SPRAYLINE_PROJECT_ROOT` 沒有設好，請重新設定。

## 六、第二步：只測 DB 連線，不寫資料

執行：

```powershell
python scripts\check_db_connection.py
```

這一步只確認：

```text
1. PostgreSQL 是否可連線
2. Database/versionB function 是否可 import
3. 基本查詢 function 是否可呼叫
```

期待看到類似：

```text
[OK] PostgreSQL connection opened
[OK] latest_batches
[OK] unack_alerts
[OK] future_summary
[OK] station_status_Station_1_demo
[OK] connection closed
```

如果這一步失敗，請先不要跑寫入測試。

常見問題：

```text
password authentication failed
-> DB_USER / DB_PASSWORD 錯

could not connect to server
-> DB_HOST / DB_PORT 錯，或 PostgreSQL 沒開

database "sprayline" does not exist
-> DB_NAME 錯，或資料庫尚未建立

relation does not exist
-> 表格尚未建立，或連到錯的 DB
```

## 七、第三步：執行 DB 回寫 smoke test

這一步會真的寫入測試資料。

執行：

```powershell
python scripts\run_db_smoke_test.py --write-test-data --station Station_1
```

```text

3. 執行 MonitoringWorker 一次
4. 判斷 threshold
5. 寫入 alert_event
6. 建立 alert_cause_link
7. 建立 alert_response_link
8. 更新 batch_station_status
9. 寫入 future_prediction_result
```

若成功，代表少榆端已完成主要 DB 回寫流程。

## 八、第四步：用 SQL 查驗結果

可用 pgAdmin 或 psql 查詢。

### 1. 查 alert_event

```sql
SELECT event_id, batch_id, station_id, sensor_name, measured_value, state, cause, ts, message
FROM alert_event
WHERE batch_id LIKE 'B_SHAOYU_E2E_%'
ORDER BY ts DESC
LIMIT 5;
```

期待結果：

```text
應看到測試 batch_id 對應的 alert_event。
cause 應為 cause_id，例如 FILTER_CLOG。
```

---

### 2. 查 alert_cause_link

```sql
SELECT acl.alert_id, acl.cause_id, acl.is_primary
FROM alert_cause_link acl
JOIN alert_event ae ON ae.event_id = acl.alert_id
WHERE ae.batch_id LIKE 'B_SHAOYU_E2E_%'
ORDER BY ae.ts DESC
LIMIT 5;
```

期待結果：

```text
應看到 alert_id 對應 cause_id。
```

---

### 3. 查 alert_response_link

```sql
SELECT arl.alert_id, arl.response_id, arl.executed_at, arl.operator_id
FROM alert_response_link arl
JOIN alert_event ae ON ae.event_id = arl.alert_id
WHERE ae.batch_id LIKE 'B_SHAOYU_E2E_%'
ORDER BY ae.ts DESC
LIMIT 5;
```

期待結果：

```text
應看到 alert_id 對應 response_id。
```

---

### 4. 查 batch_station_status

```sql
SELECT batch_id, station_id,
       robot_arm_state,
       nozzle_state,
       filter_state,
       compressor_state,
       spray_width_state,
       quality_state,
       filter_response_id,
       write_time
FROM batch_station_status
WHERE batch_id LIKE 'B_SHAOYU_E2E_%'
ORDER BY write_time DESC
LIMIT 5;
```

期待結果：

```text
應看到對應測試 batch 的 component state 被更新。
若測試是 filter 相關異常，filter_state / filter_response_id 應有值。
```

---

### 5. 查 future_prediction_result

```sql

       predicted_ok_rate,
       predicted_ng_count,
       quality_score,
       risk_level,
       model_input_source,
```

期待結果：

```text

```

## 九、測 duplicate alert suppression

少榆0616ver_4 有補重複 alert 抑制，避免 Timer 每分鐘檢查時，同一異常重複寫入。

測法：

先跑一次：

```powershell
python scripts\run_db_smoke_test.py --write-test-data --station Station_1
```

接著 5 分鐘內再跑一次 MonitoringWorker：

```powershell
python -m webservices.monitoring_worker.monitoring_worker
```

若 suppression 生效，應看到類似：

```text
skipped: True
reason: recent_unacknowledged_alert_exists
```

注意：不要用 `run_db_smoke_test.py --write-test-data` 連跑兩次測 duplicate，因為那支每次會產生新的 batch_id，不算同一異常。
